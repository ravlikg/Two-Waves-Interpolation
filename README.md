### 1. Початковий алгоритм (`NoOpenGl.cpp`)
1. **Початкові налаштування** вікна програми та створення основних фізичних змінних.
2. **Основний цикл симуляції** (працює поки вікно програми не закрите користувачем):
   * **2.1)** Прийняття вхідних даних від користувача (перевірка, чи закрив він вікно, чи натискав якісь кнопки, чи рухав повзунки налаштувань) та миттєва зміна фізичних змінних на основі цього.
   * **2.2)** Інкремент пройденого часу симуляції.
   * **2.3) Для кожного пікселя у вікні:**
     * **2.3.1)** Знайти геометричну відстань від цього конкретного пікселя до кожного з джерел хвиль.
     * **2.3.2)** Застосувати формулу гармонічних коливань для кожної хвилі окремо.
     * **2.3.3)** Додати математичні значення двох хвиль (принцип суперпозиції хвиль).
     * **2.3.4)** Розфарбувати кожен піксель: якщо отримане значення додатне (гребінь) - фарбуємо у синій колір (чим більший гребінь, тим яскравіша хвиля); якщо від'ємне (впадина) - фарбуємо у червоний.
   * **2.4)** Очищення вікна програми перед новим кадром.
   * **2.5)** Оновлення текстури зображення новими прорахованими пікселями.
   * **2.6)** Відображення всього кадру на екрані монітора.

---

### 2. Виявлена проблема та аналіз причини

* **Проблема:** Занадто повільна робота симуляції. При роздільній здатності **200x200 пікселів** програма видавала всього **45 FPS**. Спроба запустити симуляцію у Full HD призводила до повного зависання системи (менше 1 FPS).
* **Причина:** Усі математичні обчислення (дистанції, синуси) виконувалися на **центральному процесорі (CPU)** послідовно — піксель за пікселем, в один потік. Водночас у комп'ютері є **відеокарта (GPU)**, яка архітектурно створена для виконання таких паралельних завдань у тисячі разів швидше. Проте відеокарті потрібні спеціальні інструкції. Для цього пишеться код на OpenGL (а саме фрагментний шейдер), який графічний чіп транслює у власні машинні команди.

---

### 3. Як змінився наш алгоритм (з OpenGL та ImGui)

1. **Налаштування шейдера:** Тепер під час ініціалізації програми ми завантажуємо та компілюємо **фрагментний шейдер** (`wave.frag`), який компілюється безпосередньо драйвером відеокарти.
2. **Оптимізація циклу:** З основного циклу програми на CPU **повністю зник пункт 2.3** (важкий вкладений цикл по кожному пікселю). Процесор більше нічого не знає про окремі пікселі.
3. **Нова логіка роботи конвеєра:**
   * **2.3.4) [Замість циклу]:** Процесор просто бере поточні значення фізичних змінних (час `t`, амплітуди, частоти, фази та координати джерел) і за один такт передає ці 10-12 чисел у пам'ять відеокарти через команди `shader.setUniform`.
   * **2.5) Запуск шейдера:** Під час виклику команди `window.draw(sprite, &shader)` запускається наш фрагментний шейдер прямо на графічному процесорі. У ньому весь наш математичний код виконується для **усіх пікселів одночасно (паралельно)** з неймовірною швидкістю завдяки тисячам обчислювальних ядер відеокарти.

---

### 4. Фінальний результат

* **Результат оптимізації:** Після перенесення обчислень на GPU, при роздільній здатності **Full HD (1920x1080 пікселів)**, симуляція видає **3000+ FPS** на відеокарті **RTX 3060**.
